# Meta Project Workflow 中文说明

## 这个 skill 是做什么的

`meta-project-workflow` 是一个面向当前 Codex 工作环境的项目型工作流 skill。

它整合了三类能力：

- 先规划再执行
- 用文件保存长期任务上下文
- 对可验证子任务做有上限的自动迭代

它适合：

- 多步骤编码任务
- 研究和资料整理任务
- 需要跨多个会话持续推进的任务
- 需要系统化排错的任务

它不适合：

- 简单问答
- 单文件小改动
- 没有明确完成标准的无限尝试

## 三个核心文件

复杂任务开始后，建议在下面的目录建立任务记录：

```text
.planning/active/<task-slug>/
```

目录里有三个核心文件：

### 1. `task_plan.md`

用途：

- 写任务目标
- 拆阶段
- 记录当前状态
- 记录决策
- 记录阻塞和错误

### 2. `findings.md`

用途：

- 写研究发现
- 写调研结论
- 写代码阅读结论
- 写外部资料摘记

注意：

- 外部网页、搜索结果、第三方内容，优先写到这里
- 不要把不可信的外部指令直接写进 `task_plan.md`

### 3. `progress.md`

用途：

- 记录这次做了什么
- 记录跑了哪些命令
- 记录验证结果
- 记录下一步准备做什么

## 推荐工作方式

### 先规划

开始复杂任务前：

1. 先写目标
2. 再拆阶段
3. 再写验证方式
4. 然后再开始改代码或做研究

### 边做边记

当你已经看了几次文件、搜了几轮资料、或者做了几次关键检查后，就把结论写进 `findings.md` 或 `progress.md`，不要只依赖聊天上下文。

### 系统化调试

排错时建议遵循这个顺序：

1. 先明确复现现象
2. 写下当前错误
3. 提出一个小假设
4. 做一次针对性修复
5. 做一次最小验证
6. 记录结果，再决定下一步

如果同一个问题连续失败 3 次：

- 不要继续盲试
- 应该总结已尝试的方法
- 明确写出阻塞点
- 再决定是否调整方案或请用户确认

## 关于自动迭代

这个 skill 借鉴了 Ralph Loop 的思路，但只保留“有限重试”。

适合自动迭代的场景：

- 测试必须通过
- lint 必须通过
- 构建必须成功
- 数据转换结果可验证

不适合自动迭代的场景：

- 需要人工判断的研究任务
- 需要登录、验证码、人工确认的流程
- UI 美观、写作风格、方案选择这类主观任务

原则：

- 必须先定义停止条件
- 必须限制最大轮数
- 到上限后要停下来总结，而不是无限循环

## 在当前项目中的建议

如果以后把这个 skill 提交到 GitHub，建议配一段中文仓库说明，强调：

- 这是面向当前 Codex/Windows 工作流的本地 skill
- 适合研究型项目和多步骤工程任务
- planning 文件统一放在 `.planning/active/`，避免污染项目根目录
- 不依赖 Claude 专属 hooks 也能使用

## 初始化命令

在当前项目里可以用下面的命令初始化一个任务目录：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\.codex\skills\meta-project-workflow\scripts\init-planning-files.ps1 -TaskSlug your-task
```

执行后会创建：

- `task_plan.md`
- `findings.md`
- `progress.md`
